# NEXXOS MVP - Project Instructions

## Overview
NEXXOS is a Venezuelan automotive parts marketplace mobile app built with React Native + Expo.

## Tech Stack
- React Native + Expo (SDK 54), TypeScript, expo-router file-based routing
- Backend: NestJS on Node.js (deployed at EXPO_PUBLIC_API_URL)
- UI: Custom components, no heavy UI library (react-native-paper available but minimal use)

## Design System
- **Colors**: Primary #FFC107, Accent #121212, Background #FFFFFF, Section #F5F5F5, Border #E0E0E0
- **Text**: Primary #121212, Subtitle #333333, Secondary #757575
- **Status**: Success #4CAF50, Error #F44336, Info #2196F3
- **Spacing**: 8pt grid (4, 8, 16, 24, 32, 48)
- **Border Radius**: sm:8, md:12, lg:16, xl:24, full:9999

## Auth Flow
- JWT stored in expo-secure-store (mobile) / localStorage (web)
- AuthContext manages user state, auto-login on mount
- After login/signup → role-selection → client or vendor tabs

## API Integration
- Axios instance at src/services/api.ts with auto-attach Bearer token
- Services: auth, catalog, requests, vendor, chat, upload
- All API calls use optional chaining and safe defaults

## Key Patterns
- useFocusEffect for data refresh on screen focus
- Pull-to-refresh on all list screens
- Chat polling every 5 seconds
- Catalog data cached in CatalogContext
- File uploads via S3 presigned URLs

## Push Notifications
- **Backend**: NotificationModule (Global) with NotificationService using expo-server-sdk (dynamic ESM import)
- **DB**: push_tokens table linked to users (one user = multiple tokens/devices)
- **Endpoints**: POST /api/push-tokens (register), DELETE /api/push-tokens (logout)
- **5 notification events**:
  1. NEW_REQUEST → vendedores matcheados (createRequest)
  2. NEW_RESPONSE → cliente (respondToRequest)
  3. NEW_MESSAGE → otro participante del chat (sendMessage)
  4. REQUEST_CLOSED → vendedores que respondieron (closeRequest)
  5. RATING_RECEIVED → vendedor calificado (closeRequest)
- **Frontend**: Token registration in AuthContext on user login, unregister on logout
- **Navigation**: NotificationNavigator in _layout.tsx routes taps to correct screen
- **Firebase**: Requires google-services.json + service account key for Android
- **iOS**: Auto-configured during build
- app.json plugin: ["expo-notifications", {"defaultChannel": "default"}]

## Chat Image Attachments
- **Schema**: ChatMessage has `messageType` (default "text") and `imageUrl` (nullable) columns
- **Backend**: sendMessage accepts optional messageType/imageUrl; getMessages returns them
- **Frontend**: ChatMessage component renders image bubble with fullscreen preview modal
- **Chat screen**: Attach button (📎) opens Galería/Cámara menu, picks image, uploads to S3 (public), sends as image message
- **Upload flow**: expo-image-picker → getPresignedUrl(isPublic=true) → PUT to S3 → completeUpload → returns public URL → sendChatMessage with messageType=image
- **Dependencies**: expo-image (for Image component), expo-image-picker (already installed)

## Vendor Plans System
- **Plans**: Beta (unlimited, priority 2, $0, hidden), Gratuito (50/mo, priority 1, $0, visible), Pro (500/mo, priority 3, $19.99/mo), Premium (unlimited, priority 4, $49.99/mo)
- **Schema**: Plan model (slug, solicitudesMensuales, prioridad, precioMensual, precioAnual, comisionPorcentaje, visibleEnApp), VendorSubscription (estado, fechaAsignacion, fechaExpiracion, fechaGracia), VendorMonthlyRequests (vendorId, year, month, count)
- **Auto-assignment**: New vendors get Beta plan (4 months). Early vendors (pre-2026-07-01) use 2026-07-01 as reference date.
- **Expiration flow**: ACTIVE → GRACE_PERIOD (5 days) → EXPIRED (auto-assign Gratuito)
- **Request filtering**: createRequest checks canReceiveRequests() per vendor, increments monthly count after match
- **Frontend**: VendorPlanCard component on profile screen. Banners on home: ≤15 days warning, grace period alert, limit reached alert.
- **Cron**: check-expirations every 6 hours (POST /api/plans/check-expirations)
- **Endpoints**: GET /api/vendors/my-plan, GET /api/plans, GET/PATCH /api/admin/plans, POST /api/admin/vendors/:vendorId/plan, GET /api/admin/vendors-plans
- **Module**: PlansModule (plans.module.ts, plans.service.ts, plans.controller.ts) imported in AppModule, AuthModule, RequestsModule

## Vendor Response Time Metrics
- **Data source**: `RequestVendorMatch` table has `deliveredAt`, `respondedAt`, `declinedAt` timestamps
- **Endpoint**: `GET /api/vendor/response-metrics` returns avg/median/fastest/slowest response time (ms), response rate, totals
- **Dashboard**: `GET /api/vendor/dashboard` now returns `deliveredAt`, `respondedAt`, `declinedAt`, `responded`, `declined` per match
- **Home cards**: Each card shows a time label below based on status:
  - PENDING: "⏳ Sin responder · 2h 15min" (orange, live elapsed time)
  - RESPONDED: "✅ Respondida en 30min" (green)
  - DECLINED: "✖ Declinada en 1h 5min" (red)
  - CERRADA: "🔒 Cerrada · Respondida en X" / "Declinada en X" / "No respondida" (gray)
- **formatDuration()**: Helper in vendor/index.tsx converts ms → human-readable (s/min/h/d)
