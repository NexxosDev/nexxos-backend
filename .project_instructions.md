# NEXXOS MVP - Project Instructions

## Critical Architecture Decisions

### S3 Upload Strategy (IMPORTANT - June 2026)
- **Abacus hosted S3** uses temporary STS credentials (`ASIA*` prefix) that rotate frequently
- **Presigned URLs DO NOT WORK** from the mobile app because credentials expire between URL generation and mobile upload
- **Solution**: Use **proxy upload** (`POST /api/upload/direct`) — mobile sends file to backend via multipart/form-data, backend uploads to S3 with fresh credentials
- The presigned URL flow still exists for backwards compatibility but should NOT be used for mobile uploads on the Abacus dev backend
- **Direct S3 URLs** (e.g., `https://bucket.s3.region.amazonaws.com/key`) work fine for public files (reading/viewing)

### S3 Credential Resolution (CRITICAL)
- **Own AWS bucket**: `nexxos-storage-app` in `us-east-1`
- **IAM user**: `nexxos-app` with `NexxosS3Policy`
- **Env var names**: `S3_ACCESS_KEY_ID` / `S3_SECRET_ACCESS_KEY` (NOT `AWS_*` — Abacus overrides those)
- `aws-config.ts` uses a 4-tier credential resolution:
  1. **Env vars `S3_ACCESS_KEY_ID` / `S3_SECRET_ACCESS_KEY`** — permanent AKIA keys (all environments)
  2. **DB config** `API_AWS_ACCESS_KEY_ID` / `API_AWS_SECRET_ACCESS_KEY` (if prisma provided, AKIA only)
  3. **Hosted storage STS credentials** (Abacus fallback, auto-refreshed)
  4. **Bare SDK** (last resort)

### URL Resolution in Chat Messages
- Some old messages in DB have bare S3 keys (e.g., `37513/public/uploads/...`) instead of full URLs
- `resolveS3Url()` in `chat.service.ts` handles this by prepending the full S3 domain when URLs don't start with `http`

### Backend Environments
- **Dev (Abacus)**: `https://nexxos-api.abacusai.app` — uses S3_ACCESS_KEY_ID → nexxos-storage-app
- **Producción (Render)**: `https://api.nexxos.app/` — same AKIA keys, same bucket

### Cambio de Backend (PROCEDIMIENTO ESTÁNDAR)
Cuando el usuario pida cambiar de backend, SOLO hacer esto:
1. Editar `react_native_space/src/config/backend.ts` — alternar el comentario entre las dos líneas
2. `build_and_save_mobile_app_checkpoint` — rebuild y deploy
**NO hacer nada más.** No tocar .env, no tocar api.ts, no tocar login.tsx. El archivo `backend.ts` es la única fuente de verdad y alimenta tanto la conexión API como el indicador del login.
